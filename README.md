# Context Compression

Context Compression trains a language model to replace long conversation history with a much shorter summary that still preserves the information needed to continue the task.

The repository is now organized around a single implementation based on the integrated `context_compression_final` version. Older prototypes, duplicate code paths, and extra design documents are intended to be removed.

## What It Does

- Uses GRPO with Unsloth and TRL to train a summarization policy for context compression.
- Supports real dialogue datasets and synthetic dialogue generation.
- Scores outputs with a weighted reward combining:
  - task success
  - semantic similarity / reproduction
  - output format compliance
  - compression ratio
- Includes training, evaluation, and inference entry points.

## Installation

```bash
pip install -r requirements.txt
```

## Training

Train with the default config:

```bash
python train.py --config config.yaml
```

High-memory single-GPU run for a 95.6GB NVIDIA G4 environment:

```bash
python train.py --config configs/nvidia_g4_95_6gb.yaml
```

Quick smoke test with synthetic data:

```bash
python train.py --use_synthetic_data --synthetic_samples 100 --max_steps 100
```

Auto-tune config by GPU memory:

```bash
python train.py --gpu_memory_gb 24
```

## Evaluation

```bash
python evaluate.py \
  --model_path ./checkpoints \
  --test_data ./data/test.jsonl \
  --output_dir ./eval_results
```

Expected evaluation input is JSON or JSONL. Each sample should contain at least:

```json
{
  "context": "full conversation text",
  "gold_summary": "reference summary"
}
```

## Inference

Interactive mode:

```bash
python inference.py --model_path ./checkpoints --interactive
```

Direct context input:

```bash
python inference.py \
  --model_path ./checkpoints \
  --context "long conversation text"
```

File input:

```bash
python inference.py \
  --model_path ./checkpoints \
  --input_file ./context.txt \
  --output_file ./context.summary.txt
```

## Data Pipeline

The current codebase supports:

- DialogSum
- SAMSum
- UltraChat
- nohurry/Opus-4.6-Reasoning-3000x-filtered
- ianncity/KIMI-K2.5-450000x
- synthetic conversation generation

Training samples are converted into a Hugging Face dataset with:

- a chat-style `prompt`
- flattened `context`
- optional `gold_summary`
- metadata such as `sample_id` and `domain`

For the reasoning datasets, the pipeline converts rows into resumable compression contexts:

- `Opus`: `problem` becomes the user turn, `thinking` becomes the long assistant trace to compress, and `solution` is kept as the downstream answer / optional reference summary.
- `KIMI`: chat `messages` are normalized into turns; assistant responses are split into `reasoning` and `answer` segments when tags such as `<think>...</think>` are present.

## Additional Dataset Sources

These are good Hugging Face candidates to add next if you want more realistic agent and coding trajectories. They are not wired into the current loaders yet.

### Real or mostly real coding-agent traces

- [`SWE-bench/SWE-smith-trajectories`](https://huggingface.co/datasets/SWE-bench/SWE-smith-trajectories): real software-fixing trajectories collected for SWE-smith. The dataset card describes 50,000 trajectories generated by coding agents on real GitHub issues, which is a strong fit for long-horizon coding trace compression.
- [`nebius/SWE-smith`](https://huggingface.co/datasets/nebius/SWE-smith): large-scale software engineering agent trajectories. The dataset card positions it as a corpus built from 50,000 real-world GitHub issues and trajectory data for agentic SWE training.
- [`OpenHands/Devin-SWE-Bench-verified-trajectories`](https://huggingface.co/datasets/OpenHands/Devin-SWE-Bench-verified-trajectories): coding-agent trajectories aligned to SWE-bench Verified tasks. Useful when you want multi-step tool use, repo inspection, patching, and validation traces.
- [`amcodercoder/claude-code-subagent-trajectories`](https://huggingface.co/datasets/amcodercoder/claude-code-subagent-trajectories): community dataset of Claude Code style subagent trajectories. This is relevant if you specifically want traces closer to Claude Code workflows, but it is not an official Anthropic export.
- [`jondurbin/agentic-coding`](https://huggingface.co/datasets/jondurbin/agentic-coding): agentic coding interactions oriented around code tasks. This is a useful adjacent source when you want coding-oriented trajectories but not necessarily full terminal execution logs.

### Adjacent agent execution traces

- [`lydonch/hackercup-menv`](https://huggingface.co/datasets/lydonch/hackercup-menv): execution traces from a code-generation environment with action-level interaction structure. Useful if you want tool/action sequences rather than only final code diffs.
- [`open-r1/codeforces-cots`](https://huggingface.co/datasets/open-r1/codeforces-cots): coding chain-of-thought style traces for competitive programming. These are reasoning traces rather than full terminal-agent logs, but they are still useful for compression experiments on long coding contexts.

### Reasoning-trace datasets worth considering

- [`ianncity/KIMI-K2.5-450000x`](https://huggingface.co/datasets/ianncity/KIMI-K2.5-450000x): large reasoning-trace corpus distilled from `KIMI-K2.5`. The dataset card says it contains 450,000 reasoning traces, with coding making up 60% of the distribution and a total token count of 1.8B. This is a strong source for long reasoning and coding-context compression, but it is not a real terminal-agent trajectory dataset.
- [`nohurry/Opus-4.6-Reasoning-3000x-filtered`](https://huggingface.co/datasets/nohurry/Opus-4.6-Reasoning-3000x-filtered): smaller reasoning dataset with fields such as `problem`, `thinking`, `solution`, `difficulty`, and `category`, and about 2.33k rows in the viewer. This is useful for reasoning-summary experiments, but again not for real tool-use traces.

### Important caveat on the Opus dataset

- The `nohurry/Opus-4.6-Reasoning-3000x-filtered` dataset card explicitly says to prefer the original dataset because it was updated with better filtering.
- If you use that family of data, check the source linked on the card before standardizing on the filtered fork.

### Notes on Codex and Claude Code exports

- I did not find a clearly labeled public Hugging Face dataset that is obviously a raw official "Codex export" corpus.
- The Claude Code-style dataset above appears to be community-curated rather than an official export.
- For training context compression, the most defensible additions for real agent behavior are still the SWE-smith and OpenHands trajectory datasets.
- KIMI and Opus are better treated as supplementary reasoning-trace corpora rather than substitutes for real coding-agent execution traces.

## Reward Model

The default reward combines:

```text
R_total =
  0.4 * task_success +
  0.3 * reproduce +
  0.1 * format +
  0.2 * compression
```

`config.yaml` controls model, LoRA, GRPO, data, evaluation, and output settings.
